# Quality Guidelines

> 这里定义 `src-tauri/` 的默认质量门槛。

---

## Required Patterns

- 严格 TDD：先失败测试，再实现，再整理。
- 缺陷修复先补回归测试。
- 复杂功能优先拆到服务层，命令层保持薄。
- 所有功能必须能通过 CLI 模拟器执行关键路径。
- 没有 CLI 模拟器或 CLI 无法覆盖关键场景的功能，默认不允许标记为完成。

---

## Forbidden Patterns

- 为了兼容旧行为长期保留双轨逻辑，除非需求明确要求。
- 过度防御性编程：大量无证据的兜底分支、静默吞错、重复校验。
- 把性能敏感逻辑直接塞进命令装配层。
- 只依赖手工点界面验证，不补自动化验证。
- 把 CLI 模拟器当成演示脚本，而不是正式验证入口。

---

## 性能要求

- 评估算法复杂度，尤其是批量处理、扫描、排序和聚合路径。
- 控制文件 IO、系统调用和内存复制次数。
- 对热路径优先做可测量优化，不做想象式微优化。

---

## 测试与检查

默认检查项：

- `cargo test`
- `cargo fmt --check`
- `cargo clippy --all-targets --all-features`
- 与当前功能对应的 CLI 模拟命令

执行自动化测试时，默认超时时间建议不超过 `60s`，特殊场景单独说明。

---

## Code Review Checklist

- [ ] 测试是否先于实现或至少与实现同时补齐
- [ ] 是否存在应删除却保留的旧兼容逻辑
- [ ] CLI / Tauri / 服务层三者是否保持一致
- [ ] 是否能通过 CLI 完整复现主要行为，而不依赖 GUI
- [ ] 性能和资源管理是否有明显风险
- [ ] 错误路径是否清晰且可验证
