# Tauri Command Guidelines

> 前端与 Rust 层之间的边界，必须通过明确命令契约维护；CLI 模拟器与 Tauri Command 必须共用核心逻辑。没有 CLI 模拟器的功能，默认不算完成。

---

## 命令设计原则

- 每个命令都要有明确的输入类型、输出类型和错误语义。
- 命令名表达业务动作，不使用含糊的 `handle_*`、`process_*` 作为对外命名。
- 复杂功能优先拆成多个小命令，不做超大“万能命令”。

---

## CLI 模拟器硬性要求

- 所有功能必须提供 CLI 模拟入口。
- CLI 模拟器是开发、测试、调试和回归验证的核心入口，不是可有可无的辅助工具。
- 没有 CLI 模拟器的功能，一律视为未达到交付标准。
- CLI 模拟器与 Tauri Command 必须调用同一套服务层，不允许复制业务逻辑。
- 测试优先针对服务层和 CLI 层验证，再补 Tauri 命令层装配验证。

### 必须覆盖的内容

- 主成功路径
- 可预期失败路径
- 关键边界输入
- 命令契约变更后的回归场景

### 禁止事项

- 为了赶进度跳过 CLI 模拟器，只保留 GUI 路径
- CLI 和 Tauri Command 各自维护一套业务逻辑
- CLI 只能“调用一下”却无法观察关键输出或错误结果

推荐调用链：

```text
前端 invoke ─┐
CLI 调用   ─┼──> 应用服务 ──> 领域逻辑 / 基础设施
测试入口   ─┘
```

---

## 契约变更规则

当命令输入输出发生变化时，必须一并更新：

- Rust 命令签名
- 前端调用封装
- CLI 参数与输出
- 相关测试
- 文档或规范中记录的契约说明

不保留无明确需求的旧兼容命令。

---

## 错误处理

- 错误要可区分、可定位，不返回模糊字符串作为唯一信息。
- 在命令边界做必要的输入校验，在服务层维持业务约束。
- 错误模型要服务于调用方决策，而不是堆满无意义“保险分支”。

---

## 评审清单

- [ ] 命令命名是否表达真实业务动作
- [ ] 输入输出类型是否清晰且可测试
- [ ] 是否已提供可用于开发和回归的 CLI 模拟入口
- [ ] 是否复用了同一套服务逻辑
- [ ] CLI 是否覆盖成功、失败和边界场景
- [ ] 契约变更是否同步更新所有调用方
