# brainstorm: Tauri 邮箱客户端项目规划

## Goal

构建一款基于 `Tauri + React + TypeScript + Rust` 的桌面优先邮箱客户端。
这次任务的目标不是直接实现功能，而是先把产品边界、MVP、跨层架构、CLI 模拟器策略和阶段路线图规划清楚，并持续记录到 Trellis。
当前产品方向已经进一步收敛为：**面向收件与验证码处理的多邮箱统一工作台**。

## What I already know

- 当前仓库仍接近官方模板：
  - 前端只有默认 `greet` 页面
  - Rust 侧只有一个 `greet` command
- 当前技术栈已经落位：
  - 前端：`React 18 + TypeScript + Vite 6`
  - 桌面层：`Tauri 2`
  - 后端原生层：`Rust 2021`
- `src-tauri/tauri.conf.json` 里已经配置 `bun run dev` / `bun run build`。
- `.trellis/spec/backend/` 已明确给出硬规则：
  - 严格 TDD
  - 所有功能必须提供 CLI 模拟器
  - CLI / Tauri Command / 测试必须共用核心逻辑
- `.trellis/spec/frontend/` 目前还是模板化占位文档，后续需要补齐项目真实规范。
- 当前环境里没有可用的 Python 解释器，Trellis 脚本无法直接运行，因此本次按脚本约定手工完成初始化与记录。

- `.trellis/spec/guides/index.md` 引用了 `project-context.md`，但该文件当前缺失，属于规范资产缺口。

## Assumptions (temporary)

- 产品优先面向桌面端个人用户，而不是企业协同场景。
- MVP 先聚焦“收件、分类、提取、聚合”，而不是传统邮箱里“完整写信/协作”优先。
- 规划原则采用“两段式推进”：
  - 长期方向优先选择体验更强的版本
  - MVP 先做最小可用，并为增强版预留接口与演进路径
- 系统将采用明确分层：
  - React UI
  - 前端调用封装
  - Tauri commands
  - Rust 应用服务层
  - 领域逻辑 / 基础设施
  - CLI 模拟器
- CLI 模拟器会作为开发、测试、调试、回归的正式入口，而不是演示脚本。
- 首期需要本地缓存或索引能力，但可以先从单机本地方案起步。

## Requirements (evolving)

- 明确产品目标用户与核心价值。
- 明确种子用户：优先服务开发者、独立开发者、产品测试者等高频注册 / 登录 / 验证码处理人群。
- 明确 MVP 的功能边界与非目标范围。
- 明确首期账户接入策略和协议边界：MVP 先支持通用 `IMAP/SMTP + 手动配置`。
- 明确账户形态：首期即支持多账号，并以统一工作台为主体验，而不是账号切换式单账号产品。
- 明确产品方向：首期重点服务“收件友好”和“验证码/验证链接处理”场景。
- 明确发送边界：发送能力未来需要完整具备，但当前不进入 MVP 详细规划与实现优先级。
- 明确网站匹配上下文获取方式：MVP 先由用户手动输入或粘贴域名，同时预留浏览器扩展联动路线。
- 明确网站清单沉淀策略：MVP 采用半自动确认，同时预留全自动沉淀和小模型辅助判断路线。
- 明确离线边界：MVP 在线优先，但已同步邮件支持本地缓存与离线查看，完整离线能力后续演进。
- 明确提取与分类策略：MVP 规则优先，识别失败回退原始邮件展示，并预留小模型辅助接口。
- 明确“最近验证邮件”边界：MVP 只收录包含验证码或验证链接的邮件。
- 明确匹配成功后的快捷动作：MVP 提供复制验证码、打开验证链接、打开原邮件，并为浏览器插件一键打开 / 一键填入预留路线。
- 明确同步抓取策略：MVP 采用“元数据优先 + 疑似验证邮件预抓取正文”。
- 明确凭据安全策略：MVP 优先使用系统安全存储，应用本地只保存非敏感元数据。
- 明确同步触发策略：MVP 采用“手动刷新 + 固定间隔轮询”，后期演进到更实时的 IMAP IDLE / 推送。
- 明确保留窗口策略：MVP 默认同步最近 30 天邮件，并支持按需补拉更早邮件。
- 明确默认同步文件夹：MVP 首期优先同步 `Inbox + Spam/Junk`。
- 明确工作台首页默认视图：MVP 以“最近验证邮件”作为首页主视图。
- 明确最近验证邮件的生命周期：默认展示最近 48 小时，并支持手动标记已处理。
- 明确网站匹配命中策略：域名精确匹配优先，未命中时展示候选站点。
- 明确默认轮询周期：`3 分钟 + 手动刷新`。
- 明确跨层架构与目录演进方向。
- 明确 CLI 模拟器的职责、入口和覆盖范围。
- 拆分阶段性里程碑，形成可执行的实现路线图。
- 记录主要风险、依赖和待确认决策。
- 统一工作台的 MVP 至少应包含：
  - 聚合收件箱
  - 多邮箱统一查看最近验证邮件
  - 按账号筛选
  - 自动提取验证码 / 验证链接
  - 注册 / 安全 / 营销三分类
  - 当前网站验证码快速匹配
  - 注册过的网站清单

推荐采用以下基础分层：

```text

   │
   ▼

   ▼
```

### 初版模块规划

- `accounts`
  - 账户配置、鉴权信息、连接测试
  - 首期以通用 `IMAP/SMTP` 参数模型为核心，避免一开始绑定特定邮箱厂商
  - 支持多账号接入、账户启停与默认身份管理
- `mailboxes`
  - 文件夹列表、同步状态、未读计数
  - 既支持按账号查看，也支持统一工作台聚合视图
- `messages`
  - 列表、详情、正文、已读状态
  - 需要保留账号归属信息，支持跨账号聚合查询
- `extraction`
  - 从邮件主题、正文和常见模板中提取验证码、验证链接与平台名称
  - MVP 先用规则和模板识别，失败时回退原始邮件展示
- `classification`
  - 将邮件归类为注册 / 安全 / 营销
  - MVP 先以规则分类为主，后续可接入小模型辅助
- `workspace`
  - 聚合收件箱、最近验证邮件、当前网站快速匹配、网站清单
  - 首期网站上下文来自用户手动输入或粘贴域名
  - 服务层接口需要允许未来接入浏览器扩展提供的网站上下文
  - 网站清单首期通过半自动确认沉淀，后续可演进到全自动
- `compose`
  - 新建、回复、转发、发送
- `sync`
  - 首次同步、增量同步、失败重试
- `settings`
  - 本地配置、日志级别、同步策略

- CLI 输出必须可观察成功、失败和关键边界结果。
- 自动化测试优先走服务层和 CLI，再补命令层装配验证。

### 首个已确认决策

- 账户接入策略：MVP 先支持通用 `IMAP/SMTP + 手动配置`
- 原因：
  - 最容易先打穿“账户接入 -> 连接测试 -> 同步 -> 阅读 -> 发送”的完整链路
  - 更适合先定义稳定的命令契约、服务层边界和 CLI 模拟入口
  - 可以把 OAuth / 厂商适配留到后续阶段，而不是一开始就拉高跨层复杂度

### 第二个已确认决策

- 账户形态：首期即支持多账号
- 产品体验：以“统一多账户工作台”为主，而不是“切换账号查看”
- 影响：
  - 同步模型需要支持按账号独立拉取、按工作台统一聚合
  - 前端信息架构需要同时支持“统一视图”和“按账号钻取”
  - 本地缓存与索引从一开始就要带 `account_id`、`mailbox_id` 等边界字段
  - CLI 也需要支持聚合视图验证，而不只是单账号命令

### 第三个已确认决策

- 统一工作台的 MVP 深度：
  - 聚合收件箱
  - 多邮箱统一最近邮件
  - 最近验证邮件聚合视图
  - 按账号筛选
- 这样可以先做出明确的“工作台”体验，同时保持查询和索引复杂度可控

### 第四个已确认决策

- 发送能力未来也要完整具备
- 但当前阶段不作为 MVP 主轴，不进入本轮详细规划重点
- 当前先把产品与架构注意力集中在：
  - 收件
  - 同步
  - 提取
  - 分类
  - 匹配
  - 网站清单沉淀

### 第五个已确认决策

- 当前网站上下文获取方式：
  - MVP 先由用户手动输入或粘贴当前网站域名
  - 当前阶段不引入浏览器扩展
- 但服务层和命令契约需要预留浏览器扩展联动路线
- 这样可以先把“网站 -> 邮件匹配”核心链路做扎实，同时避免过早引入浏览器权限、跨进程通信和平台差异复杂度

### 第六个已确认决策

- 注册过的网站清单：
  - MVP 先采用半自动确认沉淀
  - 系统先做候选识别，再由用户确认是否加入网站清单
- 演进方向：
  - 预留全自动沉淀路线
  - 可接入小模型辅助做站点识别、置信度判断与归类建议
- 这样可以先控制误判和脏数据，同时不给未来智能化演进设限

### 第七个已确认决策

- 离线能力：
  - MVP 先走在线优先
  - 已同步邮件支持本地缓存与离线查看
  - 更完整的离线能力放到后续阶段
- 这样既保住桌面客户端的基本体验，又不会在首期过早陷入更复杂的同步一致性和本地索引问题

### 第八个已确认决策

- 总体产品策略：
  - 终局方向可以优先追求更强体验
  - 但 MVP 先做最小可用版本
  - 每个增强方向都要在第一版里预留接口，而不是等以后推倒重来
- 这条原则适用于：
  - 离线能力
  - 浏览器扩展
  - 全自动网站沉淀
  - 小模型辅助判断
  - 未来完整发送体验

### 第九个已确认决策

- 验证码提取与邮件分类：
  - MVP 先走规则优先
  - 识别失败时回退到原始邮件展示，而不是强行给出错误结果
  - 后续预留小模型辅助接口，但当前不把模型作为硬依赖
- 这样可以先把核心链路做稳定，同时控制成本、隐私和误判风险

### 第十个已确认决策

- “最近验证邮件”视图：
  - MVP 只收录包含验证码或验证链接的邮件
  - 不把所有注册 / 安全邮件一并塞进这个视图
- 这样能保持该视图足够聚焦，用户也更容易理解它的用途：快速处理验证码和验证链接

### 第十一个已确认决策

- 种子用户优先级：
  - 开发者
  - 独立开发者
  - 产品测试者
- 这些用户具备高频注册、登录、切换环境、处理验证码和验证链接的强需求
- 这意味着 MVP 应优先优化：
  - 最近验证邮件聚合
  - 当前网站快速匹配
  - 一键复制验证码 / 打开验证链接这类高频动作

### 第十二个已确认决策

- 当前网站匹配成功后的 MVP 快捷动作：
  - 复制验证码
  - 打开验证链接
  - 打开原邮件
- 后续演进：
  - 预留浏览器插件的一键打开与一键输入能力
- 这样可以先覆盖开发 / 测试用户的最高频动作，同时不把插件集成复杂度提前压进 MVP

### 第十三个已确认决策

- 邮件同步的正文抓取策略：
  - MVP 先走“元数据优先”
  - 对疑似验证邮件预抓取正文
  - 不对所有最近邮件一股脑抓取完整正文
- 这样可以在体验和成本之间取得平衡：
  - 能较快完成验证码 / 验证链接提取
  - 能控制同步流量、本地缓存体积和解析成本
  - 也给后续更激进的预抓取策略预留了空间

### 第十四个已确认决策

- 账号凭据的本地存储策略：
  - MVP 优先走系统安全存储
  - 应用自己的数据库 / 本地缓存只保存非敏感元数据
- 这样可以尽量复用各平台已有的安全能力，避免首期自己承担过重的密钥管理复杂度

### 第十五个已确认决策

- 同步触发方式：
  - MVP 先采用“手动刷新 + 固定间隔轮询”
  - 后期再演进到更实时的 IMAP IDLE / 推送方案
- 这样可以先保证验证码场景下的可用性与稳定性，同时给后续更强体验留出演进空间

### 第十六个已确认决策

- 邮件保留窗口：
  - MVP 默认同步最近 30 天邮件
  - 更早邮件通过按需补拉获取
- 这样可以明显降低首次同步和本地索引压力，同时仍然保留向历史邮件扩展的能力

### 第十七个已确认决策

- 默认同步文件夹：
  - MVP 首期优先同步 `Inbox + Spam/Junk`
- 原因：
  - 这两个文件夹最贴近验证码工作台的核心价值
  - 很多验证邮件会误入垃圾箱
  - 可以避免首期把 `Sent`、`Archive` 等噪音较大的数据一并拉进来

### 第十八个已确认决策

- 多账号工作台首页：
  - MVP 默认展示“最近验证邮件”
- 原因：
  - 最贴合产品核心价值
  - 用户一打开就能立刻处理验证码和验证链接
  - 比传统“聚合收件箱”更能体现这个产品的差异化

### 第十九个已确认决策

- 最近验证邮件的生命周期：
  - 默认展示最近 48 小时
  - 支持用户手动标记“已处理”
- 这样既保证首页足够聚焦，也能避免旧验证码长期堆积污染工作台

### 第二十个已确认决策

- 当前网站匹配策略：
  - 先做域名精确匹配
  - 未命中时给出候选站点
- 这样可以先保证结果可信度，再逐步演进到更强的模糊匹配和自动判断

### 第二十一个已确认决策

- 默认同步轮询周期：
  - `3 分钟 + 手动刷新`
- 这样在验证码场景里能兼顾一定及时性，同时避免轮询过于激进带来的资源浪费

## Risks / Concerns

- 邮件协议和提供商差异可能显著拉高首期复杂度。
- 凭据安全存储策略会影响平台兼容性与用户体验。
- 本地缓存方案会影响后续搜索、离线能力和性能。
- 如果不尽早定义 CLI 契约，后续 GUI 与测试会反复返工。
- 当前 frontend spec 仍是模板文档，后续需要补真实项目规范。
- 多账号统一工作台会显著抬高聚合查询、同步调度、状态展示与错误归因复杂度。
- 验证码提取和邮件分类在不同站点模板下会有较多边界情况。
- “当前网站快速匹配”如果依赖浏览器上下文，会引入额外集成复杂度。
- 如果手动输入的网站域名与邮件里真实站点标识不一致，需要定义模糊匹配和兜底展示策略。
- 如果未来引入小模型辅助站点判断，需要控制成本、延迟、隐私边界和可解释性。
- 即便 MVP 只做“已同步可离线查看”，也要提前定义缓存淘汰、索引粒度和同步状态提示。
- 规则提取和分类需要先定义可解释的失败回退路径，否则用户会怀疑结果可信度。
- 如果未来做浏览器插件一键填入，需要提前考虑权限边界、页面适配差异和误填风险。
- “疑似验证邮件”的判定规则如果太宽，会造成正文抓取过量；太窄又会影响提取命中率。
- 系统安全存储的跨平台差异需要在 Tauri/Rust 层尽早验证，否则会影响账户接入体验。
- 固定间隔轮询的默认周期如果设置不当，会在“及时性”和“资源消耗”之间产生明显张力。
- 30 天窗口对验证码工作台足够友好，但如果用户需要追溯更早注册记录，按需补拉体验需要足够顺滑。
- 各邮箱服务商对垃圾箱命名不一致，`Spam/Junk` 的文件夹识别规则需要单独设计。
- 48 小时窗口虽然很聚焦，但仍需给用户提供查看更早验证邮件的补充入口。
- 域名精确匹配优先可以保证可信度，但候选站点排序策略需要设计好，否则会影响体验。
