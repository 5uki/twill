# PRD UI 可执行摘要

## 目的

将当前已确认的 UI 决策收敛为可直接指导实现拆分的摘要，作为主 `prd.md` 的 UI 增补。

配套文档：

- 详细讨论与决策：[ui-notes.md](./ui-notes.md)
- 线框与结构定义：[ui-wireframe.md](./ui-wireframe.md)

## Goal

为 Twill 定义一套面向“验证码/验证链接工作台”的 MVP UI 结构，使用户能在多账号场景下快速找到、判断并处理最新验证邮件。

## In Scope

- 顶栏全局上下文与搜索
- 左侧工作台导航
- `Recent verification` 首页结构
- 高密度列表行结构
- 右侧详情区结构
- 已处理项分组策略
- 窄窗口退化
- 最小键盘交互基线

## Out of Scope

- 品牌化重视觉探索
- 完整设置页细节
- 完整命令面板系统
- 浏览器扩展 UI
- 完整传统邮箱树导航

## 已确认 UI 决策

- UI 框架使用 `@fluentui/react-components`
- UI 基调选择标准 Fluent 工作台
- 左侧导航采用产品化工作台入口，而不是传统邮箱树
- 首页默认进入 `Recent verification`
- 工作区采用三栏结构：
  - 左侧导航
  - 中间列表
  - 右侧详情
- 列表采用高密度专业列表
- 列表行信息重心为动作优先
- 每行始终显式显示账号归属
- 默认按最新时间优先排序
- 常用筛选常驻，复杂筛选进入面板
- `当前站点` 放在全局顶栏
- `当前站点` 与 `全局搜索` 分离
- 右侧详情采用提取结果优先
- 右侧快捷动作固定在顶部
- 已处理项移入次级分组
- 已处理状态采用自动与手动并存
- 窄窗口下保留左侧，右侧详情退化为 `Drawer`
- 账号与设置入口放在右上角账户区
- MVP 定义最小键盘优先交互基线

## 用户主路径

```text
输入当前站点
   ->
查看最近验证邮件
   ->
在列表中快速确认哪一封可用
   ->
在右侧执行复制验证码或打开验证链接
   ->
邮件转入已处理次级分组
```

## UI Acceptance Criteria

- [ ] 首页默认打开 `Recent verification`
- [ ] 顶栏同时提供 `当前站点` 输入与 `搜索` 入口，且二者分离
- [ ] 左侧一级导航只出现工作台入口，不出现传统邮箱树作为主入口
- [ ] 中间列表默认按时间倒序排列
- [ ] 中间列表中每行都显式展示账号归属
- [ ] 列表行可直接判断是否存在验证码或验证链接
- [ ] 右侧详情顶部固定提供高频快捷动作
- [ ] 右侧详情默认先展示提取结果，再展示原始邮件
- [ ] 已处理项不会继续停留在默认焦点区，而是进入次级分组
- [ ] 执行高价值动作后可自动标记已处理，同时支持手动撤销
- [ ] 窄窗口下详情使用 `Drawer` 承载，而不是切换到独立详情页
- [ ] 至少支持最小快捷键集：列表导航、复制验证码、打开链接、标记已处理、聚焦搜索/站点输入

## 实现切片建议

### Slice 1：应用壳层

- 建立 `FluentProvider`
- 搭建顶栏、左 rail、中间 pane、右详情 pane
- 放置账户区与同步入口

### Slice 2：首页工作台

- 实现 `Recent verification` 视图
- 建立常驻筛选条
- 完成列表分组和最新时间排序

### Slice 3：详情与动作

- 固定顶部快捷动作
- 展示提取结果
- 展示原始邮件内容
- 接通已处理状态变更

### Slice 4：响应式与键盘

- 右侧详情 `Drawer` 化
- 接入最小快捷键集
- 完成焦点管理与基础可访问性

## 风险与注意点

- 高密度列表容易在实现时被做成“信息过满”，需要严格控制字段顺序和视觉层级
- 如果右侧详情区内容过长，必须确保顶部快捷动作仍然稳定可见
- 已处理自动标记逻辑必须可撤销，否则用户会失去控制感
- 键盘快捷键要考虑 Windows 与 macOS 的差异表达
