# UI 讨论记录（2026-03-29）

## 当前结论

- UI 框架采用 `@fluentui/react-components`，即 Fluent UI React v9。
- 不采用旧的 `@fluentui/react` v8 作为新项目主线。
- UI 先服务于 MVP 的“验证邮件工作台”，不按传统邮箱客户端的信息结构直接复制 Outlook。

## 依据

- Fluent UI 官方仓库将 React Components v9 标为新的、面向未来的方向，并明确鼓励迁移到 v9。
- Fluent 2 的设计原则强调：
  - 自然适配平台
  - 为专注而构建
  - 一致且可访问
- 这些原则和 Twill 当前产品方向是对齐的：
  - 桌面优先
  - 高频处理验证码/验证链接
  - 需要明显的信息层级与快速动作

## 对 Twill 的约束映射

- 技术栈是 `Tauri + React 18 + TypeScript`，因此应优先选择 Fluent Web React 方案，而不是 WinUI 原生路线。
- 产品是跨平台桌面应用，视觉可以偏 Windows，但不能把交互完全绑死在 Windows 特性上。
- 当前核心价值是“快读、快筛、快操作”，所以 UI 应该优先保证：
  - 扫描效率
  - 键盘可达性
  - 低干扰
  - 明确的主次操作层级

## 推荐的 MVP UI 结构

```text
+-------------------------------------------------------------+
| Top bar: workspace switch / search / sync / account / theme |
+-------------------+----------------------+-------------------+
| Left rail         | Center primary pane | Right detail pane |
|                   |                      |                   |
| - Recent verify   | - Verification list | - Message detail  |
| - Unified inbox   | - Unified inbox     | - Quick actions   |
| - By account      | - Filters / states  | - Site matching   |
| - Sites           |                      |                   |
+-------------------+----------------------+-------------------+
```

- 默认首页不是“收件箱”，而是 `Recent verification`。
- 左侧负责“视图切换”，中间负责“候选项列表”，右侧负责“详情 + 快捷动作”。
- 在窄宽度下退化为双栏：
  - 左 rail 保留
  - 详情区改为 `Drawer`

## Fluent 组件建议映射

- 应用壳层：
  - `FluentProvider`
  - `Toolbar`
  - `Button`
  - `Input`
  - `SearchBox` 或输入框组合
  - `Avatar`
  - `Badge`
- 视图切换：
  - 左侧主导航优先自定义轻量 rail
  - 小范围内容切换使用 `TabList`
- 邮件列表：
  - MVP 优先“高信息密度列表行”
  - 不建议一开始就上重型表格心智
  - 如需多列排序，再评估 `DataGrid/Table`
- 详情与设置：
  - 账号配置、筛选条件、站点确认流优先用 `Drawer`
  - 关键确认动作使用 `Dialog`
- 状态反馈：
  - 非阻断反馈用 `Toast`
  - 加载态用 `Spinner` / `Skeleton`

## 视觉方向建议

- 主基调采用 Fluent 2 的简洁分层，不做大面积强品牌化改造。
- 字体优先跟随 Fluent 体系，桌面端可接受 Segoe 风格，但不要手动做重字体定制。
- 表面材质建议：
  - MVP 先以 `solid` 为主
  - `acrylic` 只用于轻量浮层
  - `mica` 只作为后续增强方向，不纳入第一版 UI 基线
- 理由：
  - 先保证跨平台一致性
  - 避免 Tauri 窗口材质能力过早侵入组件层设计

## 三种可行方案

### 方案 A：标准 Fluent 工作台（推荐）

- 以 Fluent v9 标准组件和 token 为主。
- 布局采用三栏 / 双栏响应式工作台。
- 视觉偏克制，强调扫描效率和快捷操作。

优点：

- 落地最快
- 与 Fluent 官方组件契合度最高
- 后续补测试和重构成本更低

代价：

- 个性化和“桌面原生感”不会特别强

### 方案 B：Windows 风格强化版

- 在方案 A 基础上增强窗口材质、层次和微软桌面感。
- 可能引入更多 `mica/acrylic` 风格表达和更强的顶部窗口感。

优点：

- 品牌识别更强
- 更像 Windows 桌面工具

代价：

- 平台差异处理更复杂
- Tauri 窗口层能力会更早进入架构讨论

### 方案 C：Fluent 基础上的品牌化改造

- 使用 Fluent token、可访问性和基础控件。
- 外层导航、卡片、列表行做较强定制。

优点：

- 可以做出更独特的产品识别
- 更适合把“验证码工作台”做成差异化产品

代价：

- 自定义 UI 维护成本更高
- 更容易偏离 Fluent 现成的交互语义

## 当前建议

- 先选方案 A 作为 MVP 基线。
- 给方案 B 预留窗口材质升级空间。
- 在方案 A 内允许少量方案 C 风格的“产品化列表行”和“快捷动作卡片”。

## 已确认决策

### 决策 1：UI 基调采用标准 Fluent 工作台

- 用户已选择方案 A。
- MVP 以 Fluent UI React v9 的标准组件、token 和工作台布局为主。
- 第一版不追求强 Windows 材质风格，也不追求重品牌化外观。

原因：

- 更符合当前阶段“先把工作流做顺”的目标。
- 能优先验证信息架构、列表密度、快捷动作和状态反馈。
- 后续如需增强 Windows 感或品牌化表达，可以在不推翻基础组件体系的前提下迭代。

### 决策 2：左侧导航采用产品化工作台导航

- 左侧导航不以传统邮箱文件夹树作为第一层入口。
- 第一层入口优先放产品视图：
  - `Recent verification`
  - `Unified inbox`
  - `By account`
  - `Sites`
- 传统账号/文件夹结构不消失，但放到次级层级或进入对应视图后再展开。

原因：

- 更符合 Twill 当前的产品定位，而不是做一个普通邮箱壳。
- 用户的高频任务是“找到最近需要处理的验证邮件”，不是先浏览邮箱树。
- 这样也更容易把统一工作台、多账号聚合和站点匹配做成清晰的一级心智。

### 决策 3：主工作区采用三栏常驻详情

- 整体结构固定为：
  - 左侧工作台导航
  - 中间消息列表
  - 右侧消息详情与快捷动作
- 右侧详情在桌面主视图中常驻，不采用“进入单独详情页”作为主路径。
- `Drawer` 只作为窄宽度或局部配置流的退化方案。

原因：

- 更适合验证码/验证链接场景下的连续处理。
- 用户可以保持列表上下文，不需要反复进入和返回详情页。
- 复制验证码、打开验证链接、查看原邮件这些动作天然适合在右侧详情区集中承载。

### 决策 4：中间主列表采用高密度专业列表

- `Recent verification` 和 `Unified inbox` 的主列表以高信息密度为优先目标。
- 一行一项，不采用大卡片式布局。
- 优先支持快速扫读、连续选择和键盘操作。

原因：

- Twill 当前更接近“高频处理工作台”，而不是内容消费型邮箱阅读器。
- 验证邮件场景下，用户最需要的是在有限空间里快速比较：
  - 来自哪个站点
  - 属于哪个账号
  - 是否识别出验证码/验证链接
  - 是否值得立刻处理
- 高密度列表也更适合后续加入批量状态、快捷键和筛选维度。

### 决策 5：列表行采用动作优先的信息重心

- 列表行优先突出“可立即执行的结果”，而不是完整邮件内容摘要。
- 每行应优先暴露的信息包括：
  - 是否识别出验证码
  - 是否识别出验证链接
  - 站点/平台名称
  - 账号归属
  - 时间状态
- 发件人、主题、正文摘要保留，但降到次级层级。

原因：

- 当前产品的主要目标不是帮助用户“读懂所有邮件”，而是帮助用户“尽快完成下一步动作”。
- 在验证码场景里，真正决定效率的是：
  - 能否第一眼判断这封是否有用
  - 能否马上复制或打开
  - 能否少点一次进入详情的成本
- 这也与高密度列表和三栏工作台的整体方向一致。

### 决策 6：右侧详情区采用提取结果优先

- 右侧详情区默认先展示：
  - 验证码
  - 验证链接
  - 站点匹配结果
  - 快捷动作
- 原始邮件正文、头信息和其他内容放在后续区块中。
- 原始内容仍然必须易于查看，但不作为默认视觉重心。

原因：

- Twill 的目标是帮助用户尽快完成验证动作，而不是先阅读邮件全文。
- 当提取成功时，最有价值的信息就是“现在可以执行什么”。
- 当提取失败时，再顺滑退回到原始邮件内容查看，能形成清晰的主路径和兜底路径。

### 决策 7：右侧详情区的快捷动作固定在顶部

- `复制验证码 / 打开验证链接 / 打开原邮件` 这类高频动作固定在右侧详情区顶部。
- 这些动作不跟随正文内容滚动，不采用悬浮工具条作为主方案。
- 提取结果和原始邮件内容在其下方按区块展开。

原因：

- 这是当前产品里最高频、最明确的一组操作。
- 顶部固定可以减少滚动和来回定位成本。
- 与“提取结果优先”的详情策略相互配合，能形成稳定的视觉与操作锚点。

### 决策 8：常用筛选常驻，复杂筛选进入面板

- 中间列表上方常驻一组高频筛选入口。
- 推荐常驻的筛选项包括：
  - `全部`
  - `未处理`
  - `有验证码`
  - `有链接`
  - `当前站点`
- 更复杂的条件组合放入筛选面板，使用 `Drawer` 或 `Dialog` 承载。

原因：

- 高频筛选必须一眼可见，不能让用户每次都多点一步。
- 同时也要避免把高密度工作台做成一整排拥挤控件。
- 这种分层方式与 Fluent 工作台结构更匹配，也便于后续扩展更多高级条件。

### 决策 9：“当前站点”上下文输入放在全局顶栏

- `当前站点` 作为整个工作台的全局上下文存在，放在顶栏而不是某个局部面板里。
- 用户可以随时输入、粘贴或替换域名。
- 顶栏中的站点上下文会影响：
  - `Recent verification` 视图中的 `当前站点` 筛选
  - 站点匹配结果优先级
  - 相关快捷动作和候选站点提示

原因：

- “我现在在处理哪个站点”是一个跨列表、跨详情、跨视图的全局状态。
- 放到顶栏后，整个工作台的行为会更一致，用户也更容易理解当前过滤上下文。
- 这也是 Twill 区别于传统邮箱客户端的重要产品心智。

### 决策 10：顶栏中“当前站点”和“全局搜索”分离

- 顶栏保留两个语义明确的入口：
  - `当前站点` 输入
  - `全局搜索` 入口
- 不采用单一输入框同时承担“搜邮件”和“设置站点上下文”两种职责。
- 如后续需要高级入口，可额外增加命令式统一入口，但不替代这两个基础控件。

原因：

- 两者的意图完全不同：
  - `当前站点` 是工作上下文
  - `全局搜索` 是查询行为
- 分离后，用户更容易理解当前界面为什么被过滤、排序或匹配。
- 这也更符合当前 MVP 以清晰、稳定、高频处理为主的目标。

### 决策 11：列表中每行始终显式显示账号归属

- 在统一工作台中，每条邮件都必须直接显示其账号归属。
- 账号归属不是只放到详情区，也不是仅做非常弱的提示。
- 推荐在高密度列表中使用稳定、紧凑的账号标识形式，例如：
  - 账号名
  - 邮箱地址缩略
  - 轻量 badge

原因：

- 多账号是 Twill 的核心前提，不应让用户在来源判断上产生额外认知成本。
- 验证邮件场景里，用户经常需要立刻判断“这是哪个账号收到的”。
- 显式显示账号归属能减少误用验证码、误开链接和误读上下文的风险。

### 决策 12：“最近验证邮件”默认按最新时间优先排序

- `Recent verification` 视图默认按邮件接收时间倒序排列。
- 最新收到的验证邮件始终出现在更靠前的位置。
- 不采用“可执行结果优先”或更复杂的混合排序作为默认规则。

原因：

- 用户最常见的真实任务是“我刚刚触发了一次注册或登录，现在在等最新那封邮件”。
- 时间优先的规则最直观，几乎不需要学习成本。
- 对 MVP 来说，这个排序最稳定，也更容易和未处理状态、站点过滤、账号归属一起形成可预期的行为。

### 决策 13：已处理项移入次级分组，而不是直接消失

- 当一封验证邮件被视为“已处理”后，它不再停留在默认焦点列表中。
- 但它也不直接从当前页面完全消失，而是移入次级分组，例如：
  - `已处理`
  - 列表底部的次级区块
- 默认主视图继续聚焦“待处理项”，同时保留用户回看刚处理内容的能力。

原因：

- 这比“立即移除”更可追踪，用户不容易因为误操作丢失上下文。
- 这也比“原位变灰”更干净，不会持续污染默认焦点区。
- 对验证码工作台来说，这是兼顾效率和可回看的折中方案。

### 决策 14：“已处理”状态采用自动与手动并存

- 常见高价值动作可以自动将邮件标记为“已处理”，例如：
  - 复制验证码
  - 打开验证链接
- 同时保留手动标记与手动撤销入口。
- 自动标记不应是不可逆黑盒，用户必须能看见并改回。

原因：

- 纯手动会增加高频场景下的操作负担。
- 纯自动又容易让用户失去控制感。
- 自动与手动并存最适合当前工作台定位：默认高效，但保留可修正性。

### 决策 15：窄窗口下保留左侧导航，右侧详情退化为 Drawer

- 在桌面主宽度下维持三栏布局。
- 当窗口宽度不足以舒适容纳三栏时：
  - 左侧工作台导航继续保留
  - 中间列表保持主视图
  - 右侧详情区改为 `Drawer`
- 不采用“小窗口直接跳单页”的主退化策略。

原因：

- 这样能最大限度保持工作台的稳定心智。
- 用户在窄窗口下仍然可以快速浏览列表，不会失去主操作流。
- 右侧详情天然适合以抽屉形式临时展开和关闭。

### 决策 16：账号与设置入口放在右上角账户区

- 顶栏右上角放统一的账户区入口。
- 账户区负责承载：
  - 当前账户入口
  - 多账号管理入口
  - 设置入口
- 设置不作为左侧一级导航项。

原因：

- 左侧一级导航应保留给工作台核心视图，而不是系统管理项。
- 账户与设置属于全局辅助能力，更适合放在顶栏右上角。
- 这也符合常见桌面工作台产品的视觉惯例。

### 决策 17：MVP 定义最小键盘优先交互基线

- 第一版就定义最小快捷键集，不等到后期再补。
- 最小集合建议覆盖：
  - 上下选择列表项
  - 打开或关闭详情
  - 复制验证码
  - 打开验证链接
  - 标记已处理 / 撤销已处理
  - 聚焦搜索或站点输入
- 不追求第一版就做完整的命令系统，但要保证高频动作可走键盘。

原因：

- 目标用户包含开发者和测试用户，他们对键盘效率敏感。
- 既然产品定位是高频工作台，键盘路径不应完全缺席。
- 先定义最小集合，后续再扩展，会比后补更稳定。

## MVP UI 总结

### 设计基线

- UI 框架：`@fluentui/react-components`
- 视觉路线：标准 Fluent 工作台
- 核心目标：高频验证码处理，而不是传统邮箱阅读

### 页面骨架

```text
+---------------------------------------------------------------+
| Top bar                                                       |
| [Current Site] [Search]                [Sync] [Account/Settings]
+-------------------+------------------------+-------------------+
| Left rail         | Center list            | Right detail      |
|                   |                        |                   |
| Recent verify     | High-density rows      | Fixed actions     |
| Unified inbox     | Latest first           | Extracted results |
| By account        | Common filters         | Original message  |
| Sites             | Processed sub-group    |                   |
+-------------------+------------------------+-------------------+
```

### 关键行为

- 首页默认进入 `Recent verification`
- 默认按最新时间倒序
- 列表高密度、一行一项、动作优先
- 每行显式显示账号归属
- 常用筛选常驻，复杂筛选进入面板
- 右侧详情提取结果优先，快捷动作固定在顶部
- 已处理项移入次级分组
- 已处理状态支持自动与手动并存

### 响应式退化

- 宽窗口：三栏常驻
- 窄窗口：左侧保留，右侧详情改 `Drawer`

### 仍可后续再细化的点

- 具体列表行字段顺序
- `Sites` 视图的交互细节
- 设置页信息架构
- 快捷键命名与平台差异

## 当前状态

- UI 主骨架已基本收敛到可进入线框或实现前设计阶段。
- 如无新增方向变更，下一步应产出：
  - 首页线框
  - 列表行结构定义
  - 右侧详情区结构定义
